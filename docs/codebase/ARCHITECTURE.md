## Layer Organization

The project is organized **by feature/resource**, not by architectural layer:

- **Frontend**: context-per-domain (`AuthContext`, `MessageContext`, `RoomContext`) with co-located components
- **Backend**: resource-per-file routes (`auth.mongo.ts`, `conversations.mongo.ts`, etc.) registered under `/api/<resource>`

## Context Provider Stack (Frontend)

Three providers wrap the app in a fixed order:

```
AuthProvider
  в””в”Җ MessageProvider
       в””в”Җ RoomProvider
            в””в”Җ <App UI />
```

- `AuthProvider` manages JWT token lifecycle and user identity
- `MessageProvider` owns all conversation/message state; depends on auth
- `RoomProvider` owns Socket.io connection and real-time room state; depends on auth

## Request Lifecycle (REST)

Example: user fetches conversations

```
1. Component calls apiClient<ConversationList>('/conversations')
2. apiClient reads auth_token from localStorage вҶ’ sets Authorization header
3. Vite proxy (dev) / nginx (prod) forwards to Express :3001/api/conversations
4. Express router: authenticateToken middleware
   в”ңв”Җ Extracts Bearer token вҶ’ verifies JWT
   в””в”Җ Looks up user in MongoDB users collection вҶ’ attaches req.user
5. Route handler: checks req.user.role
   в”ңв”Җ admin вҶ’ no filter (all conversations)
   в””в”Җ user  вҶ’ filter: is_visible=true OR shared with user
6. MongoDB query вҶ’ returns cursor вҶ’ mapped to response JSON
7. Component receives typed result
```

## MongoDB Collection Design

| Collection      | Purpose                                             | Key Indexes                                      |
|-----------------|-----------------------------------------------------|--------------------------------------------------|
| `users`         | User accounts with roles                            | `username` (unique)                              |
| `conversations` | LinkedIn conversation metadata                      | `is_visible + last_message_date`, `last_message_date` |
| `messages`      | Individual messages per conversation                | `conversation_id + date`, full-text on `content + from_name` |
| `bookmarks`     | User-saved messages                                 | `user_id`, `user_id + message_id` (unique)       |
| `shared_chats`  | Conversation shares between users                   | `to_user_id + created_at`, `conversation_id`     |
| `read_rooms`    | Collaborative reading rooms with participant state  | `code` (unique), `creator_id`, `participants.user_id` |

## Authentication Architecture

```
Login вҶ’ POST /api/auth/login
  вҶ’ bcrypt.compare(password, passwordHash)
  вҶ’ jwt.sign({ userId }, JWT_SECRET, { expiresIn: '24h' })
  вҶ’ stored in localStorage as 'auth_token'

Every subsequent request:
  вҶ’ Authorization: Bearer <token>
  вҶ’ authenticateToken middleware
      вҶ’ jwt.verify(token, JWT_SECRET)
      вҶ’ db.users.findOne({ _id: userId })
      вҶ’ req.user = { id, username, role }
```

## Admin vs User Access Pattern

A consistent inline check appears throughout all route handlers:

```ts
const isAdmin = req.user?.role === 'admin'
const filter = isAdmin ? {} : { is_visible: true }
```

Admins see all data; users see only visible records or records explicitly shared with them.

## Singleton Pattern (Backend)

`getDB()` returns the single `MongoClient`/`Db` instance created at startup. All routes import and call `getDB()` вҖ” no new connections are created per request.

## Key Design Decisions

| Decision                       | Rationale (inferred from code)                                          |
|--------------------------------|-------------------------------------------------------------------------|
| Dual file suffix (`.mongo.ts`) | Migration from SQLite to MongoDB; legacy `.ts` files kept but inactive  |
| Socket.io for rooms only       | Scroll sync requires low-latency push; REST is used for all CRUD        |
| Connection count map           | Prevents marking user offline when they have multiple browser tabs open |
| Room auto-delete when empty    | Rooms are transient; no persistent room history needed                  |
| Shared chats collection        | Allows non-admin users to view specific conversations shared by admins  |
