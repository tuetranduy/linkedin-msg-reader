## Security Risks

### 🔴 Insecure JWT Secret Default
`auth.mongo.ts` and `mongodb.ts` both fall back to a hardcoded plaintext secret:
```ts
const JWT_SECRET = process.env.JWT_SECRET || 'linkedin-msg-secret-key-change-in-production'
```
If `JWT_SECRET` is not set in the environment, tokens can be forged by anyone who knows the default.
- **Mitigation**: Throw an error at startup if `JWT_SECRET` is not set in production (`NODE_ENV === 'production'`).
- Evidence: `server/src/middleware/auth.mongo.ts`, `server/src/db/mongodb.ts`

### 🔴 Default Admin Credentials Seeded at Startup
Every time the database starts fresh, an admin account `admin / admin123` is created if none exists.
```ts
console.log('Created default admin user: admin/admin123')
```
This password must be changed immediately after first deploy. There is no enforced password rotation or first-login prompt.
- Evidence: `server/src/db/mongodb.ts`

### 🟡 JWT Token in localStorage
`auth_token` stored in `localStorage` is vulnerable to XSS. `dompurify` mitigates risk for rendered message content, but any injected script in other parts of the UI could exfiltrate the token.
- [ASK USER] Is switching to `httpOnly` cookies an acceptable change?

### 🟡 No Rate Limiting
Login endpoint (`POST /api/auth/login`) has no brute-force protection (no `express-rate-limit`, no account lockout).
- Evidence: `server/src/routes/auth.mongo.ts`

### 🟡 Wildcard CORS on Express Middleware
`app.use(cors())` with no options allows all origins on the Express HTTP layer. Socket.io has explicit origin allowlisting, but REST API does not.
- Evidence: `server/src/index.ts`

## Technical Debt

### 🟡 Dual Implementation Files (SQLite + MongoDB)
Every route and middleware exists in two versions: legacy `*.ts` (SQLite) and active `*.mongo.ts`. The legacy files are not imported anywhere but remain in the repository, creating confusion:
- `server/src/routes/auth.ts`, `conversations.ts`, `messages.ts`, `bookmarks.ts`, `search.ts`, `users.ts`
- `server/src/middleware/auth.ts`
- `server/src/db/database.ts`
- `server/data/linkedin-msg.db*`
- **Action**: Delete legacy SQLite files to reduce maintenance surface.

### 🟡 Duplicate Password Hash Fields
MongoDB `users` documents store both `passwordHash` and `password_hash`:
```ts
await db.collection('users').insertOne({
  passwordHash,
  password_hash: passwordHash,   // both written
  ...
})
```
This inconsistency risks divergence if one field is updated without the other.
- Evidence: `server/src/db/mongodb.ts`

### 🟡 `CURRENT_USER` Hardcoded in Types File
```ts
export const CURRENT_USER = {
  name: 'Tue Tran',
  profileUrl: 'https://www.linkedin.com/in/tuetranduy'
}
```
Hardcoded identity in `src/types/message.ts` means the app is tied to one LinkedIn account's export. Any other user's CSV would not correctly identify "current user" messages.
- Evidence: `src/types/message.ts`
- [ASK USER] Should this be configurable (e.g., set by admin on upload, or stored per user)?

## Performance Concerns

### 🟡 MongoDB Read Per Scroll Event
`room:scroll` socket handler performs a `db.collection('read_rooms').findOne()` on every scroll event to verify `can_control` permission before updating and broadcasting. High-frequency scroll events will generate many DB reads.
- **Mitigation**: Cache permission in memory (e.g., a `Map<socketId, boolean>` maintained on join/permission-change events).
- Evidence: `server/src/socket/handlers.ts`

### 🟡 Large Root Component (`App.tsx` — 860 lines)
`App.tsx` combines routing logic, auth gate, layout rendering, and modal state. It is the second highest-churn file. Split into smaller route components would reduce risk of merge conflicts and improve testability.

### 🟡 Large Context File (`MessageContext.tsx` — 713 lines)
Highest-churn file in the repo. Contains scroll sync logic, pagination, bookmarks, search, and conversation selection in a single context. Consider splitting into `ConversationContext` and `MessageContext`.
