## Database

### MongoDB
- **Driver**: `mongodb` native Node.js driver (not Mongoose)
- **Connection**: singleton via `connectDB()` in `server/src/db/mongodb.ts`
- **Default URI**: `mongodb://localhost:27017/linkedin-msg`
- **Production URI**: injected via `MONGODB_URI` environment variable
- **Indexes**: created once at startup in `connectDB()`
- **Auth**: username/password in connection string (see `server/.env.example`)
- Evidence: `server/src/db/mongodb.ts`, `server/.env.example`

### Legacy SQLite (inactive)
- Files exist at `server/data/linkedin-msg.db*` and `server/src/db/database.ts`
- Not mounted in production containers; MongoDB is the authoritative store
- Evidence: `server/Dockerfile` (no SQLite data volume), `server/src/index.ts` (imports only MongoDB routes)

## Real-Time Communication

### Socket.io
- **Server**: `socket.io` ^4.8.x in backend
- **Client**: `socket.io-client` ^4.8.3 in frontend
- **Transport**: WebSocket with HTTP long-polling fallback
- **Auth**: JWT token passed in socket handshake `auth.token`; validated by `authenticateSocket` in `server/src/socket/auth.ts`
- **Rooms**: Socket.io rooms named `room:${code}` for broadcast isolation
- **CORS origins**:
  - `http://localhost:5173` – `http://localhost:5176` (dev)
  - `https://linkedin.tuetran.dev` (prod)
- Evidence: `server/src/index.ts`, `server/src/socket/auth.ts`, `server/src/socket/handlers.ts`

## Authentication & Security

### JWT
- Library: `jsonwebtoken` ^9.x
- Secret: `process.env.JWT_SECRET` (falls back to insecure default — must be set in production)
- Expiry: `24h`
- Storage: `localStorage` key `auth_token` (client-side)
- Evidence: `server/src/middleware/auth.mongo.ts`, `server/.env.example`

### Password Hashing
- Library: `bcrypt`, salt rounds: `12`
- Field names in MongoDB: `passwordHash` and `password_hash` (both stored for compatibility)
- Evidence: `server/src/db/mongodb.ts` (admin seed), `server/src/routes/auth.mongo.ts`

### HTML Sanitization
- `dompurify` ^3.2.5 used on the frontend to sanitize message content before rendering
- Evidence: `package.json` dependencies

## Content Processing

### CSV Import
- Library: `papaparse` ^5.5.3
- Used to parse LinkedIn message export CSV files
- Frontend: `src/utils/csvParser.ts` (client-side CSV parsing utility)
- Backend: `/api/messages/upload` endpoint accepts multipart CSV upload
- Evidence: `package.json`, `src/utils/csvParser.ts`, `src/types/message.ts` (RawMessage interface)

## Infrastructure

### Nginx
- **Dev container**: `nginx.conf` served as `default.conf` in frontend container
- **VPS**: `nginx/linkedin.tuetran.dev.conf` — handles SSL termination, reverse proxy to containers
- Proxies `/api/*` and `/socket.io/*` to backend container
- Evidence: `nginx.conf`, `nginx/linkedin.tuetran.dev.conf`, `Dockerfile`

### Docker
- Two-service architecture: `frontend` (nginx) and `backend` (Node.js)
- Network: `linkedin-msg-network` (bridge)
- Persistent volume: `linkedin-msg-data` mounted at `/app/data` in backend
- Evidence: `docker-compose.prod.yml`

### GitHub Actions
- Deployment pipeline in `.github/workflows/deploy.yml`
- Secrets required: `VPS_HOST`, `VPS_USERNAME`, `VPS_SSH_KEY`, `VPS_PORT`, `JWT_SECRET`, `MONGODB_URI`
- Evidence: `.github/workflows/deploy.yml`
